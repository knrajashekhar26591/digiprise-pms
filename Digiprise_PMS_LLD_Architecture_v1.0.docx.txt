1. Executive Summary
1.1 Purpose
This document defines the Low-Level Design (LLD) and System Architecture for a Jira-equivalent Project Management System (PMS) embedded natively within the 
Digiprise
 CRM platform. The system is architected to deliver enterprise-grade agile project management capabilities, supporting Scrum, Kanban, and hybrid methodologies at scale for 1,000+ concurrent active users with sub-2-second response times.
The specification covers functional requirements, database design, backend service architecture, UI/UX screen layouts, security, DevOps pipelines, and non-functional requirements at production grade.

3.2 Issue Hierarchy
3.2.1 Hierarchy Levels
The system enforces a strict four-level hierarchy:
Level 1 — EPIC: Large feature or initiative. Cannot have a parent. Contains Stories.
Level 2 — STORY: User-facing requirement or feature slice. Parent = Epic.
Level 3 — SUBTASK: Atomic work unit. Parent = Story. Assigned to single dev.
Level 4 — SUB-SUBTASK: Granular breakdown for complex tasks. Parent = Subtask.

3.4 Workflow Engine
3.4.1 Workflow Structure
A workflow is a directed graph of Statuses connected by Transitions. Each project is bound to a workflow scheme that maps issue types to workflows.
Statuses: system-defined categories (To Do, In Progress, Done) with custom names
Transitions: named edges between statuses (e.g., 'Start Progress' from To Do → In Progress)
Validators: pre-conditions checked before transition executes
Post-Functions: side effects executed after transition (e.g., set Assignee, trigger webhook)
Conditions: user role or field-value constraints limiting who can trigger a transition

5. System Architecture
5.1 High-Level Architecture Overview
The system follows a layered, service-oriented architecture with the following primary tiers:
Presentation Tier: React.js SPA served via CDN / Nginx static hosting
Application Tier: ASP.NET Core 8 Web API, horizontally scalable behind load balancer
Cache Tier: Redis Cluster for session, query cache, rate limiting
Search Tier: Elasticsearch cluster for full-text issue search
Persistence Tier: SQL Server (primary), with read replicas for reporting queries
Message Tier: RabbitMQ for async job processing (notifications, webhook delivery)
Real-time Tier: ASP.NET Core 
SignalR
 for live board updates and notifications

9. Security Requirements
9.1 Authentication
JWT Bearer: HS256-signed access tokens (15-min TTL) + RS256 option for enterprise
Refresh token rotation: each use issues new token, old token invalidated
Multi-factor authentication (TOTP / authenticator app) — optional per tenant
OAuth 2.0 / OIDC: Google Workspace, Microsoft Entra ID (Azure AD) SSO via 
OpenIdConnect
 middleware
Brute force protection: account locked after 5 failed attempts; exponential backoff on subsequent unlocks
9.2 Authorization
RBAC enforced at API layer via [
Authorize(
Policy = ...)] attributes on controllers
Resource-based authorization: 
IAuthorizationHandler
 validates per-issue and per-field rights
Multi-tenant isolation: 
TenantId
 injected into every query via EF Core global filter — impossible to query cross-tenant data
9.3 Data Encryption
In-transit: TLS 1.3 enforced; HSTS header; certificate pinning optional
At-rest: SQL Server TDE (Transparent Data Encryption) for database files
Attachments: AES-256 encrypted at storage layer (Azure Blob Storage server-side encryption)
Sensitive fields (tokens, webhooks): encrypted in DB using application-level AES-256 with key from Azure Key Vault / environment-managed secret store
9.4 API Security
Input sanitization: 
HtmlSanitizer
 on rich text fields to prevent stored XSS
CORS: strict whitelist of allowed origins; no wildcard
CSP headers: restrictive Content-Security-Policy preventing inline script injection
SQL injection prevention: parameterized queries only via EF Core — no string-interpolated SQL
File upload: MIME type validation, virus scan via 
ClamAV
 integration, 25 MB limit enforced at middleware
Dependency scanning: 
Dependabot
 / 
Snyk
 in CI pipeline for known CVE alerts
